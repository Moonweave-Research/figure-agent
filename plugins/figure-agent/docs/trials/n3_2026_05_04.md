**Unexpected finding**: 0-macro N=2 is cleaner than 5-macro N=1 after iteration. Root cause: macro-internal rendering generates its own clashes (BandDiagram CB/VB text-on-path, BellCurve side curve labels). Hand-written code avoids these by design. Implication: macro library benefit is structural recognition baseline (author gets BandDiagram shape "for free"), not visual clash reduction.

## New macro candidates from Run 2 (7, all cross-cutting)

1. **`\PanelCard`** — rounded-rect background + panel-separator layout
2. **`\Cantilever`** — parametric Bezier beam clamped at one end with curvature control
3. **`\ClampBlock`** — fixed-support/boundary-condition symbol
4. **`\ElectrodeBlock`** — labeled 2D rectangular electrode with voltage annotation
5. **`\GhostOutline`** — dashed prior-state path with "prev." label
6. **`\MechanismCallout`** — colored annotation box with text (cross-cutting to any multi-panel figure)
7. **`\ForceArrow`** — styled directional force arrow (Stealth, color-coded by sign)

None of these overlap with Run 1's candidates (BandSideCurve, RowFrame, LabeledConnector, PowerLawLine/DebyeReference). All 11 combined candidates are cross-cutting — no physics-specific bias except `\BandSideCurve`.

## Implications for α / γ decision (N=2 data)

- **α (narrow L3, freeze)**: still not supported. 0% macro coverage is workable BUT "workable" ≠ "no value". Missing structural primitives (cantilever, electrode, clamp) forced ~40 lines of hand-craft per panel; a small library of cross-cutting primitives would halve this.
- **γ (continue L3 physics-specific macros)**: not the right direction. The 11 new candidates across N=1+N=2 are ALL cross-cutting mechanism/geometry primitives, not physics-specific (BandDiagram-class). Growing the physics macro library further before the mechanism/geometry library adds the wrong kind of value.
- **Emergent recommendation**: if L3 work resumes, priority shift from "physics-specific" (BandDiagram API gaps 1/2/5) to **"cross-cutting geometry/layout primitives"** (PanelCard, Cantilever, ForceArrow).

## New macro candidates from Run 3 (5, all cross-cutting geometry/layout)

1. **`\CircularInset`** — circle + internal grid/structure + border frame (used for microstructure "zoom" callouts)
2. **`\BeakerVessel`** — 2D beaker outline with fill gradient for liquid, precipitate blob
3. **`\ProcessFlow`** — horizontal arrow + text label flow connector (generic across any process diagram)

None overlap with N=1 or N=2 candidates. Across all three runs, **16 total macro candidates** identified, all cross-cutting (geometry, layout, mechanism primitives). **Only 1 physics-specific candidate** (`BandSideCurve`).

## Implications for α / γ decision (N=3 data)

- **α (narrow L3, freeze macro library)**: supported by N=3 evidence. Zero-collision-on-first-compile shows that careful hand-writing beats macro-heavy drafting. Macro library (currently 5 physics-specific) is neither a quality bottleneck nor a time-saving lever in these trials.
- **γ (continue L3 push)**: **redirected direction**. The 16 cross-cutting candidates (PanelCard, Cantilever, CircularInset, BeakerVessel, etc.) show where macro library would add genuine value, but current macro library is 100% physics-specific (trap-energy, band diagrams, Debye plots). Growing that further is the wrong priority.
- **Architecture reset confirmed**: N=3 data validates yesterday's decision (Option A, L3 hold). WARN-driven iteration + Style Lock + baseline macro library (as-is) are the real quality kernel levers. Briefing automation + Inkscape polish are the next value moves.

## Caveat — groundtruth quality in Run 2

Mid-run, user noted that the Run 2 reference figure (`fig4a_actuation_sequence.png`) has physics/geometry errors (cantilever curvature wrong, side-view perspective unclear). This was included in trial document but does not invalidate Run 2 data — the macro findings and WARN iteration results stand regardless. Run 2 is a valid "low-ground-truth-quality" datapoint showing that figure-agent produces reasonable output even from flawed input.

## Overall N=3 trial conclusion

Three independent figure types (trap-energy physics, actuator mechanism, material processing), three distinct intent diversities, three different macro coverage profiles (5 → 0 → 1), three different zero-collision convergence rates (2 iter → 1 iter → 0 iter). Data across N=1/N=2/N=3 strongly supports **Option A: L3 freeze + architectural reset toward briefing/Inkscape**.
