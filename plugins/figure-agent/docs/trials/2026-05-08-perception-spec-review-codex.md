**2. BLOCKING — The proposed topology algorithm does not actually prove it catches the six Panel A chains**

The TeX splits every polysulfide chain into three independent draw paths: linker, solid wave, faded tail: [fig1_overview_v2.tex](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/examples/fig1_overview_v2/fig1_overview_v2.tex:73). Those paths are nested under shifted and rotated scopes: [fig1_overview_v2.tex](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/examples/fig1_overview_v2/fig1_overview_v2.tex:92). PyMuPDF does return transformed final PDF coordinates, so the scope transform itself is not the problem. The problem is path segmentation.

A local PyMuPDF probe on `examples/fig1_overview_v2/build/fig1_overview_v2.pdf` found 169 drawing paths, 144 stroke endpoints, and 72 endpoints farther than 0.20 cm from any other endpoint. Filtering amber strokes gave only 4 lonely amber tail endpoints, not the spec’s expected 6, because two inward chain termini are within ~0.12-0.19 cm of neighboring chain segment endpoints. Widening the threshold to 0.50 cm would catch more “should connect” cases but would also worsen the already-large false-positive surface. The spec’s `ε=0.20cm` is not validated by the fixture it uses as the headline case: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:152).

There is also a coordinate-space gap. The TeX source uses a `14cm x 9cm` canvas then `\resizebox{178mm}{!}`: [fig1_overview_v2.tex](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/examples/fig1_overview_v2/fig1_overview_v2.tex:2), [fig1_overview_v2.tex](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/examples/fig1_overview_v2/fig1_overview_v2.tex:19). PyMuPDF returns PDF-space points after scaling and standalone border. The spec’s sample panel bbox `[0, 5.00, 3.50, 4.00]` is source-TikZ space: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:172). No mapping is specified.

**3. BLOCKING — Numerical claims are under-sourced or contradicted by probes**

`page.get_drawings()` does return vector graphics as dictionaries, but not in the friendly native schema shown by the spec. Official docs say `items` is a list of draw commands, and the API returns tuple op codes like `("l", Point, Point)` or `("c", ...)`, not named `moveto/lineto/curveto` dicts: PyMuPDF docs lines 1763-1795 and drawing example lines 231-239. The spec’s schema is implementable, but not “returned natively”: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:142).

Compile-time estimates are mixed. On this machine, current full compile for `fig1_overview_v2` took ~7.55s, while a direct `lualatex` rerun with the required `TEXINPUTS` took ~4.46s. So the 4-6s wireframe rerender estimate is plausible, but the stated current baseline `~12-15s` did not match this fixture: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:481).

The false-positive claim is not traced. The spec expects ≤2 spurious lonely flags on `golden_trap_depth_picture`: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:509). A local naive PyMuPDF endpoint probe on the golden PDF found 78 endpoints farther than 0.20 cm. That does not prove the final tuned algorithm must fail, but it does prove the spec has not earned the ≤2 claim.

**4. BLOCKING — Dependency/license story is wrong**

The spec says PyMuPDF is MIT licensed: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:572). Official PyPI says PyMuPDF is open-source GNU AGPL v3 or commercial licensed, not MIT: PyPI licensing lines 652-657. PyPI does show broad wheels for macOS, Linux, musl, and Windows, so platform availability is plausible: PyPI wheel lines 1275-1307. But the license claim must be fixed before adding it as a required dependency.

The repo currently has only `numpy`, `pillow`, `pyyaml`, and `scipy` as runtime deps: [pyproject.toml](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/pyproject.toml:6). Also, “No new TeX rendering needed” is only true for extraction. The same spec adds a wireframe lualatex rerender: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:242).

**6. BLOCKING — Scope is too large for a spike**

The spec proposes 4 YAMLs, 4 render outputs/sets, and a report in one new ~400-line script: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:62), [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:366). That is too much before proving the core question: can build geometry reliably flag Panel A’s broken network?

The MVP should be smaller: one extractor, one topology report, one debug PNG at most, using `fig1_overview_v2` as the RED case and `golden_trap_depth_picture` plus one quantitative plot as false-positive controls. `edge.png` is especially weak: it is a user-debugging artifact and overlaps heavily with wireframe for the stated chain-connectivity problem. The agent critique path benefits most from a concise topology summary and a single annotated view, not three extra images by default.

But then the acceptance criteria must be treated as trial criteria, not shipped guarantees. Right now the spec mixes “advisory WARN-tier findings” with hard-sounding claims like “Panel A should produce 6+ lonely endpoints”: [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:293), [architecture-v0.4-build-perception-pack.md](/Users/choemun-yeong/workspace/ResearchOS/[figure-agent]/plugins/figure-agent/docs/architecture-v0.4-build-perception-pack.md:501).

Fix the PyMuPDF license/dependency claim, define the source-cm ↔ PDF-cm mapping, prove a topology MVP catches the actual six-chain case without exploding false positives, and remove claims that semantic awkwardness is mechanically solved by geometry alone.
