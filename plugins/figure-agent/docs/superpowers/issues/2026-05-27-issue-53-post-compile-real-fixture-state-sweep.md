# Issue 53 — Post-Compile Real Fixture State Sweep

Status: implemented; pending commit

## Problem

Issue 50 verified the dry-run state surface without executing compile/export or
critique commands. After Issue 52, the next useful plugin check is a
post-compile sweep: compile representative real fixtures, then confirm
`/fig_drive --mode authoring|review|polish|release` still routes to one
conservative next action without stale UX hints.

The concrete risk is not rendering correctness. The risk is operator confusion:
status, driver, closeout, and audit-evidence summaries may disagree about the
next command once build-side reports exist.

## Scope

In scope:

- Run all published real fixture specs through `scripts/compile.sh`.
- Re-run `fig_driver.py` in authoring, review, polish, and release modes.
- Record the source-only and post-compile routing matrix.
- Fix narrow UX/contract defects if the sweep exposes a misleading next action.
- Keep generated build artifacts ignored and uncommitted.

Out of scope:

- Refreshing host-vision critiques.
- Editing fixture `.tex` source.
- Exporting, accepting, forcing golden state, or mutating publication
  provenance.
- Changing release, polish, or critique gate semantics beyond the misleading
  audit-evidence next-action fixes.

## Findings

The source-only checkout behaved as expected: all 8 fixtures and all 4 modes
returned `run_compile` with `first_blocker=render_missing`.

After compiling the 8 fixtures, the routing matrix was broadly consistent:

- Reference-grounded fixtures with stale or legacy critiques routed
  review/polish/release to `/fig_critique`.
- Fixtures with `critique_state=NOT_REQUIRED` routed review/polish/release to
  export.
- Authoring mode stopped after render-fresh completion and did not advance into
  critique/export/loop/polish.
- `fig1_overview_v2_pair_001_vault` surfaced the expected Issue 52
  `LP001`/`LP002` label-path candidates.

Two audit-evidence UX defects were found:

1. A fixture with no `critique.md` returned
   `evaluation_state=not_applicable` but still suggested `/fig_critique`.
   That is misleading for fixtures where critique is legitimately
   `NOT_REQUIRED`.
2. Missing, malformed, or stale `build/audit_crops/manifest.json` suggested
   `/fig_compile`, even though audit crops are generated by the critique brief
   path, not by compile.

## Fix

`scripts/audit_evidence_summary.py` now:

- leaves `next_action` empty when `critique.md` is absent and audit evidence is
  not applicable;
- routes missing/malformed/stale audit-crop manifests to `/fig_critique <name>`
  instead of `/fig_compile <name>`.

## Acceptance Criteria

- Missing critique without an audit evidence surface does not suggest host
  critique from `audit_evidence` itself.
- Missing/malformed/stale audit-crop manifests suggest `/fig_critique`, not
  `/fig_compile`.
- The 8-fixture x 4-mode post-compile matrix has no command failures.
- Generated build artifacts are not committed.
- Targeted and full verification pass.
