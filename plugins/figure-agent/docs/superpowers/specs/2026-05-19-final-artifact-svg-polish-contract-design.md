# Final Artifact SVG Polish Contract Design

**Date:** 2026-05-19
**Status:** design ready

## Purpose

`figure-agent` has enough critique, loop, export, and acceptance machinery to
find many figure defects, but the final manuscript artifact still has a gap:
some visual polish is not realistic to finish in raw TikZ alone. The missing
contract is a controlled final-artifact layer where a human or outer agent may
polish the generated SVG while preserving the TikZ source as the semantic
source of truth.

This design does not turn `figure-agent` into an SVG editor. It defines how a
polished SVG can become the declared final artifact without losing provenance,
freshness, critique closure, or publication safety.

## Current State

The current layer model is:

- L0 external inputs and references
- L2 briefing, spec, style lock, and golden contract
- L3 semantic TikZ authoring
- L4 compile, collision, clash, drift, and perception checks
- L4.5 host-orchestrated critique and adjudication
- L5 generated export artifacts
- L6 accepted/golden validation
- L7 status
- L8 reproducibility

`final_ready` is currently a compatibility alias for `release_ready`.
`/fig_export` regenerates exports from the build PDF and blocks stale critique
unless explicitly skipped. `/fig_loop` is verify-only and may recommend one
patch target, but it does not edit source, exports, accepted state, or golden
contracts.

The gap is after L5: the generated SVG can be a valid reproducible export while
still needing optical micro-polish before submission. If that polish happens
outside the repo, the loop loses the artifact, the hashes, the toolchain, the
human decision, and the distinction between visual polish and semantic change.

## Design Decision

Add a future L5.5 final-artifact layer:

```text
L3 semantic TikZ source
  -> L4 compile/perception gates
  -> L4.5 critique/adjudication/loop gates
  -> L5 generated exports
  -> L5.5 declared final artifact, optional polished SVG
  -> L6 accepted/golden/publication validation
```

TikZ remains the semantic source of truth. A polished SVG is a presentation
artifact. It may become the declared final artifact only when a manifest proves
which generated export it was derived from, which source hashes it depends on,
which classes of edits were made, and whether any edit requires backporting to
TikZ, briefing, or critique.

This should be implemented inside `figure-agent`, not as a separate plugin,
because acceptance, status, export freshness, and publication provenance are
already owned by `figure-agent`. A separate SVG editor could exist later, but
the contract and gates belong here.

## Artifact Layout

For a fixture named `<name>`:

```text
examples/<name>/
  exports/<name>.svg
  polish/
    <name>.polished.svg
    svg_polish_manifest.yaml
    svg_polish_audit.md
```

`exports/<name>.svg` remains generated by `/fig_export`. It must not be edited
by hand. `polish/<name>.polished.svg` is the optional final visual source.
`svg_polish_manifest.yaml` is the machine-readable contract.
`svg_polish_audit.md` is the human-readable closeout.

Do not duplicate `exports/<name>.svg` into `polish/` unless a future issue
finds a concrete review need. The manifest records the generated SVG hash, so
the base export is already identifiable.

## Declaration

Polished SVG must be opt-in. A fixture declares it in `spec.yaml`:

```yaml
final_artifact:
  kind: polished_svg
  manifest: polish/svg_polish_manifest.yaml
```

If `final_artifact` is absent, the fixture uses the current generated-export
path and status should report `final_artifact_state: NONE`. A stray
`polish/svg_polish_manifest.yaml` may be warned about, but it must not change
release semantics unless `spec.yaml` opts in.

This opt-in prevents an accidental draft SVG or half-written manifest from
changing acceptance behavior.

Allowed `final_artifact.kind` values:

- `generated_export` — explicit spelling of today's default generated artifact
  behavior.
- `polished_svg` — opt into Layer 5.5 polished SVG validation.

If `final_artifact.kind` is absent, status should behave as `generated_export`
for compatibility and report `final_artifact_state: NONE`.

## Manifest Schema

Schema name:

```yaml
schema: figure-agent.svg-polish-manifest.v1
fixture: <name>
base:
  source_set_hash: sha256:<hash>
  source_tex_hash: sha256:<hash>
  briefing_hash: sha256:<hash>
  spec_hash: sha256:<hash>
  generated_svg_hash: sha256:<hash>
  export_pdf_hash: sha256:<hash>
  critique_hash: sha256:<hash>
polished:
  path: polish/<name>.polished.svg
  polished_svg_hash: sha256:<hash>
  audit_hash: sha256:<hash>
  editor: human | external_tool | agent_assisted
  toolchain:
    - name: <tool name>
      version: <version or unknown>
  edit_classes:
    - label_micro_position
    - stroke_polish
    - icon_detail
    - spacing_balance
  semantic_change_declared: false
  backport_required: false
provenance:
  reviewer: <human name or accountable role>
  reviewed_at: <ISO-8601 timestamp>
  notes: <short note>
```

Required properties:

- `schema` must exactly equal `figure-agent.svg-polish-manifest.v1`.
- `fixture` must match the fixture directory name.
- `spec.yaml.final_artifact.manifest` must point to this manifest before the
  polished SVG can become release-relevant.
- all hashes must use the repo's existing `sha256:<hash>` convention.
- `base.source_set_hash` must be computed from the final-artifact input set,
  not only from `<name>.tex`, `briefing.md`, and `spec.yaml`.
- `polished.path` must point inside `examples/<name>/polish/`.
- `polished.path` must not point to generated exports, build artifacts,
  accepted exports, another fixture, or a path outside the fixture.
- `polished.audit_hash` must match `polish/svg_polish_audit.md`.
- `polished.editor` must be one of `human`, `external_tool`, or
  `agent_assisted`.
- `polished.edit_classes` must be a non-empty list of bounded values.
- `semantic_change_declared: true` or `backport_required: true` prevents final
  acceptance until the semantic source and critique loop are refreshed.
- `provenance.reviewer` and `provenance.reviewed_at` are required before a
  polished SVG can satisfy accepted-mode publication gates.

Unknown future mapping fields may be preserved, but they must not bypass the
required fields above.

Allowed `polished.edit_classes` values:

- `label_micro_position`
- `leader_line_micro_position`
- `stroke_polish`
- `icon_detail`
- `spacing_balance`
- `color_opacity_polish`
- `typography_cleanup`
- `export_cleanup`

Any edit that does not fit one of these classes should default to
`backport_required: true` until a later issue adds a bounded class with tests.

The final-artifact input set should include at least:

- `examples/<name>/<name>.tex`
- `examples/<name>/briefing.md`
- `examples/<name>/spec.yaml`
- `styles/polymer-paper-preamble.sty`
- `coordinate_hints.yaml`, when present
- `authoring_contract.md`, when present
- `authoring_plan.md`, when present
- `theory_guard.md`, when present
- `subregion_iteration_log.md`, when present
- `reference/reference_pack.md`, when present
- declared figure-level and panel-level reference images, when present

Issue 7A should reuse or mirror `scripts/quality_manifest.py` path sorting and
repo-relative hash payload conventions so this set does not drift from existing
critique freshness behavior.

## Allowed SVG-Only Edits

These edits can stay in the polished SVG when the manifest declares no
semantic change:

- optical label nudges that preserve the intended target
- leader-line endpoint micro-adjustments that preserve the target identity
- stroke cap, join, dash, and line-weight normalization
- icon edge cleanup that preserves component identity
- small spacing, balance, and white-space corrections
- color or opacity polish that preserves material identity and contrast
- typography kerning or text-outline cleanup that preserves wording

These edits are presentation changes. They do not authorize changing the
scientific story, component topology, mechanism, reference interpretation, or
accepted state.

## Must-Backport Changes

The following cannot live only in polished SVG:

- adding, removing, renaming, or re-targeting a scientific component
- changing material identity or label meaning
- changing mechanism arrows, force/field/current direction, or charge signs
- changing panel order, panel role, or storyline
- changing scale/proximity relationships that affect interpretation
- adding missing apparatus parts that critique marked as structural defects
- fixing a finding whose root cause is in briefing, spec, or TikZ source
- any edit the human reviewer cannot confidently classify as visual-only

When a must-backport change appears, the closeout path is:

1. patch TikZ, briefing, or spec as needed;
2. rerun compile/export;
3. refresh critique when freshness requires it;
4. recreate or update adjudication;
5. regenerate the polish manifest from the new generated SVG;
6. rerun final-artifact validation.

## Freshness States

Future status should expose an explicit final-artifact state:

| State | Meaning |
|---|---|
| `NONE` | no polished final artifact is declared in `spec.yaml`; generated export remains the artifact |
| `MISSING` | a polished artifact is declared but required file is absent |
| `INVALID` | manifest, path, schema, hash, or provenance is malformed |
| `STALE` | manifest hashes no longer match current source, generated export, critique, or polished SVG |
| `BLOCKED` | manifest and hashes are valid, but semantic change or backport is declared |
| `FRESH` | manifest, hashes, provenance, and polished SVG all match |

Hash freshness should be content-based, not mtime-only, and should use the
existing `scripts/quality_manifest.py` hash conventions where possible. A
polished SVG is stale when any recorded base hash differs from the current
fixture source set, generated SVG, export PDF, critique, or polished SVG
content.

`svg_polish_audit.md` should also be freshness-checked by content hash. Without
that, a human could update the manifest while leaving stale audit prose behind.

## Validation Algorithm

Future validation should follow this order:

1. Read `spec.yaml`.
2. If `final_artifact.kind` is absent or `generated_export`, return
   `final_artifact_state: NONE` and keep current generated-export behavior.
3. If `final_artifact.kind` is not `polished_svg`, return `INVALID`.
4. Resolve `final_artifact.manifest` relative to the fixture root.
5. Reject the manifest path if it escapes the fixture, points into `exports/`,
   points into `build/`, or points outside `polish/`.
6. Load and validate the manifest schema and required fields.
7. Reject if `manifest.fixture` differs from the fixture directory name.
8. Reject if `polished.path` is missing, escapes the fixture, points into
   generated exports/build artifacts, or does not live under `polish/`.
9. Recompute the current final-artifact source-set hash, generated export,
   critique, polished SVG, and audit hashes with the repo's `sha256:<hash>`
   convention.
10. Return `STALE` when any recorded hash differs.
11. Return `INVALID` when `semantic_change_declared` or `backport_required`
    has a non-boolean value.
12. Return `BLOCKED` when `semantic_change_declared: true` or
    `backport_required: true`.
13. Return `FRESH` only when the manifest is valid, paths are contained,
    required files exist, hashes match, semantic backport is not required, and
    provenance fields are present.

Acceptance-mode validation should still fail a `FRESH` polished artifact when
publication compliance, theory guard, reference pack, golden contract, or
`accepted: true` gates fail.

## Command-Facing Behavior

### `/fig_export`

`/fig_export` remains the generated-export command. It must not overwrite,
delete, or mutate `polish/<name>.polished.svg`. If future code detects that a
generated SVG changed while a polish manifest exists, it should report the
final artifact as stale through status or a dedicated validation command rather
than trying to merge SVG edits.

### `/fig_status`

`/fig_status` should eventually report:

- `final_artifact_state`
- `final_artifact_kind`: `generated_export` or `polished_svg`
- `final_artifact_path`
- freshness or validation notes

The existing `workflow_ready` can remain tied to generated source/export loop
closure. `golden_ready`, `release_ready`, and `final_ready` should only depend
on a polished SVG when the fixture declares polished SVG as the final artifact.

### `/fig_loop`

`/fig_loop` remains verify-only. After compile, critique, adjudication, export,
and major quality axes are closed, it may recommend a single SVG polish
handoff. It must not edit SVG itself.

If a polish manifest is stale or invalid, `/fig_loop` should surface that as a
status action. If `semantic_change_declared` or `backport_required` is true,
the loop should force semantic backport before recommending final acceptance.

### Accepted and Golden Gates

Accepted-mode validation should eventually distinguish two paths:

- generated final artifact: current export and golden artifact behavior
- polished SVG final artifact: polished SVG manifest, audit, hashes,
  provenance, and publication safety are required

`accepted: true` remains a human approval. The presence of a polished SVG never
sets acceptance automatically.

## Human Gates

Human review is required for:

- choosing polished SVG as the declared final artifact;
- declaring `semantic_change_declared: false`;
- marking `backport_required: false`;
- final publication provenance and disclosure;
- setting `accepted: true` or `submission-safe: true`.

Routine stale hash repair, missing manifest repair, or generated export
refresh can be agent-actionable, but semantic classification and acceptance
promotion remain human accountable decisions.

## Non-Goals

- Build an SVG editor.
- Auto-polish SVG paths.
- Import polished SVG edits back into TikZ automatically.
- Replace TikZ as the source of semantic truth.
- Mark fixtures accepted automatically.
- Mutate generated exports by hand.
- Infer scientific semantic preservation from SVG path geometry alone.
- Add external API calls or remote vision services.
- Require polished SVG for every fixture.

## Implementation Slices

### Issue 7A: SVG Polish Manifest Schema

Build parser, writer, validator, and freshness helpers for
`svg_polish_manifest.yaml`. This should be TDD and independent of status,
export, and accepted gates.

### Issue 7B: Final Artifact Status State

Expose `final_artifact_state`, `final_artifact_kind`, and validation notes in
`/fig_status` without changing existing workflow readiness for ordinary
fixtures.

### Issue 7C: SVG Polish Audit and Handoff Docs

Document the human or outer-agent handoff for producing one polished SVG and
closing it out through manifest, audit, compile/export, critique freshness, and
loop rerun.

### Issue 7D: Accepted Gate Integration

Make accepted-mode validation require polished SVG freshness and provenance
only when the fixture declares polished SVG as the final artifact.

### Issue 7E: Loop Surfacing

Teach `/fig_loop` to surface stale/invalid/fresh final-artifact state and to
recommend exactly one next action without auto-editing.

## Review Notes

### Review 1: Source of Truth and Semantic Drift

Finding: the design is acceptable only if TikZ remains the semantic source and
SVG polish cannot silently carry science changes. The must-backport list and
human gates address this. Defect found and fixed: the first draft did not say
where polished SVG is declared. The design now requires explicit
`spec.yaml.final_artifact` opt-in.

### Review 2: Freshness and State Ambiguity

Finding: mtime-based freshness would be too weak because SVG polish may happen
outside the normal compile/export cadence. The design requires content hashes
for source, generated export, critique, polished SVG, and audit prose. Defect
found and fixed: the first draft did not call out audit prose freshness. The
design now requires an audit hash or equivalent content check.

### Review 3: Acceptance and Provenance Risk

Finding: a polished SVG could otherwise be mistaken as manuscript-ready merely
because it exists. The design keeps `accepted: true` and `submission-safe: true`
as human decisions and requires reviewer provenance. Defect found and fixed:
the manifest example included audit hash after Review 2, but the required-field
list did not explicitly bind it to `svg_polish_audit.md`.

### Review 4: Integration Scope

Finding: changing `/fig_export` to manage polished SVG would create overwrite
risk and merge semantics the plugin cannot prove. The design keeps export
generated-only and moves polish validation to status/final-artifact gates.
Defect found and fixed in this hardening pass: the active architecture overview
did not mention the planned L5.5 layer, so future agents could miss the design.
The architecture overview now includes a planned Layer 5.5 section.

### Review 5: Product Direction and Modularity

Finding: a separate SVG tool could be useful later, but the acceptance contract
must live with `figure-agent` because it depends on source hashes, critique,
export freshness, golden gates, and publication provenance. The design leaves
editing capability out of scope. Defect found and fixed in this hardening pass:
the initial schema text did not enumerate `final_artifact.kind` or
`edit_classes` values. Both are now bounded in the design.

### Review 6: State Machine Completeness

Finding: the first validation algorithm had no state for valid, hash-fresh
polish that declares semantic change or required backport. Treating that as
`INVALID` would conflate malformed input with a real but blocked artifact.
Fix: add `BLOCKED` as a final-artifact state and make loop/accepted-gate issues
handle it explicitly.

### Review 7: Source-Set Freshness Completeness

Finding: the manifest example tracked `source_tex_hash`, `briefing_hash`, and
`spec_hash`, but that misses style lock, coordinate hints, reference packs,
theory guards, sub-region logs, and declared reference images. Fix: add
`base.source_set_hash` and require Issue 7A to use the repo-relative manifest
hash convention from `scripts/quality_manifest.py`.
